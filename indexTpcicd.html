Le projet en lui-meme n'est PAS le livrable principal

Le projet doit etre suffisamment complexe pour justifier toutes ces couches — un simple CRUD avec 2 routes ne suffira pas.

1 Code & Design Patterns

Evalue sur /4 points

Pourquoi c'est la partie la plus importante

Si votre code est mal structure, tout le reste s'effondre. Des fonctions de 200 lignes sont impossibles a tester unitairement. Un code sans separation des responsabilites produit des tests d'integration fragiles. Un code sans injection de dependances oblige a des mocks bricolees. La qualite de vos tests est le reflet direct de la qualite de votre code.

Architecture & structure attendues

  • Architecture claire et coherente — MVC, Clean Architecture, Hexagonal, ou toute architecture que vous pouvez justifier. L'important est que la structure soit identifiable et qu'il y ait une raison derriere.
  • Separation des responsabilites — Chaque module/classe/fichier a un role unique. Pas de logique metier dans les controleurs. Pas de requetes SQL dans les routes. Pas de console.log("ca marche") en guise de logging.
  • Code propre — Fonctions courtes, nommage explicite, pas de magic numbers, pas de code duplique (DRY). Le code doit etre lisible sans commentaires.

Rappel — Principes SOLID

  • Single Responsibility : une classe = une responsabilite
  • Open/Closed : ouvert a l'extension, ferme a la modification
  • Liskov Substitution : un sous-type doit etre substituable a son type parent
  • Interface Segregation : plusieurs interfaces specifiques > une interface generale
  • Dependency Inversion : dependre des abstractions, pas des implementations

Design Patterns — Minimum 3, a bon escient

Attention

Les patterns doivent etre utilises la ou ils resolvent un vrai probleme, pas plaques artificiellement pour cocher une case. Si un pattern complique le code sans apporter de valeur, c'est pire que pas de pattern du tout. Pour chaque pattern, vous devez justifier : ou il est utilise, pourquoi ici, et en quoi il facilite les tests ou la maintenance.

Creational

Singleton

Garantir une instance unique d'un objet. Cas d'usage : connexion BDD, logger, configuration globale.

exemple conceptuel
// Au lieu de creer 10 connexions BDD :
const db = Database.getInstance()
// Toujours la meme instance partout
Creational

Factory / Abstract Factory

Creer des objets sans specifier la classe exacte. Cas d'usage : generation de rapports (PDF/CSV/Excel), types de notifications.

exemple conceptuel
const report = ReportFactory.create('pdf')
// Facile a tester : on peut creer un FakeReport
Behavioral

Strategy

Algorithmes interchangeables a l'execution. Cas d'usage : calcul de prix, validation, tri, authentification multi-providers.

exemple conceptuel
// Changer d'algo sans modifier le code appelant
pricing.setStrategy(new StudentDiscount())
pricing.calculate(cart)
Behavioral

Observer

Notification automatique lors d'un changement d'etat. Cas d'usage : evenements, webhooks, notification quand une ressource change.

exemple conceptuel
orderService.on('created', sendEmail)
orderService.on('created', updateStock)
// Decoupler les effets secondaires
Structural

Repository

Abstraction de l'acces aux donnees. Cas d'usage : changer de BDD sans toucher a la logique metier, faciliter le mocking dans les tests.

exemple conceptuel
// En prod : PostgresUserRepo
// En test : InMemoryUserRepo (Fake)
const service = new UserService(userRepo)
Structural

Decorator

Ajouter des comportements sans modifier la classe. Cas d'usage : logging, caching, validation autour d'un service existant.

exemple conceptuel
// Ajouter du cache sans toucher au service
const service = new CachedService(
  new LoggedService(baseService)
)
Structural

Adapter

Rendre compatible une interface existante avec une autre. Cas d'usage : wrapper d'API externe, migration de librairie.

exemple conceptuel
// Wrapper une API externe pour l'isoler
const mailer = new SendGridAdapter(config)
// Si on change de provider : un seul fichier
Behavioral

Middleware / Chain of Responsibility

Pipeline de traitement sequentiel. Cas d'usage : auth → validation → rate limiting → handler.

exemple conceptuel
app.use(authMiddleware)
app.use(validateMiddleware)
app.use(rateLimitMiddleware)
// Chaque couche est testable independamment

Le lien patterns ↔ tests

Un bon design pattern rend le code testable par construction. Le pattern Repository permet de remplacer une vraie BDD par un Fake en test. Le pattern Strategy permet de tester chaque algorithme isolement. Le pattern Adapter permet de mocker une API externe proprement. Si vos patterns ne facilitent pas vos tests, c'est qu'ils sont mal utilises.

2 Tests

Evalue sur /3 points

Rappel — La pyramide des tests

E2E (5-10%)
Lent, fragile, peu nombreux
Integration (15-20%)
Interaction entre modules
Unitaires (70-80%)
Rapides, stables, nombreux

Obligatoire Tests unitaires

  • Couvrent la logique metier : calculs, transformations, validations
  • Structure AAA systematique : Arrange (preparer) → Act (executer) → Assert (verifier)
  • Nommage explicite : should [resultat] when [condition]
  • Un test = un comportement — pas de test qui verifie 5 choses a la fois
  • Mocking uniquement pour les dependances externes (API, BDD, filesystem)

Rappel cours : Si mocker vos dependances est complique, c'est que votre code est mal decouplé. Un pattern Repository ou Adapter resout ce probleme.

Obligatoire Tests d'integration

  • Testent l'interaction entre modules (routes + controleurs + BDD)
  • Base de test : SQLite en memoire, conteneur Docker, ou base dediee
  • Cas nominaux ET cas d'erreur (404, 400, 401, 409...)
  • Testent les middlewares (auth, validation, rate limiting)

Si applicable Tests end-to-end

  • Simulent un vrai utilisateur (Cypress, Playwright, Selenium)
  • Minimum 2-3 scenarios critiques si vous avez un front
  • Si pas applicable (API pure, CLI), expliquez pourquoi et compensez avec plus de tests d'integration

Couverture de code

70%

Seuil minimum global

Bloquant dans le pipeline

Qualite

> Quantite

Un test sans assertion = 0 valeur

Rapport

Genere dans le pipeline

Artefact ou badge dans le README

Rappel — Les pieges du cours

  • 100% couverture ≠ 0 bugs — Visez des tests de qualite, pas du remplissage
  • Test sans assertion — Un test qui appelle une API sans verifier la reponse passe toujours mais protege de rien
  • Tests flaky (instables) — Meme code, meme resultat a chaque fois. Pas de dependance a l'heure, au reseau, ou a l'ordre d'execution
  • Pyramide inversee — Beaucoup d'E2E et peu d'unitaires = suite lente et fragile

3 Qualite de code automatisee

Evalue sur /2 points

Linting & Formatting

  • Linter configure avec des regles personnalisees et justifiees (pas juste la config par defaut)
  • Formatter configure (Prettier, Black, etc.)
  • Pre-commit hook (Husky, pre-commit) : lint + format avant chaque commit
  • Le pipeline CI verifie aussi (double filet de securite)
Linters par langage
JavaScript/TS → ESLint + Prettier
Python → Ruff ou Flake8 + Black
Java → Checkstyle + SpotBugs
PHP → PHP_CodeSniffer + PHPStan
Go → golangci-lint
Rust → Clippy + rustfmt

Analyse statique & SonarCloud

  • SonarCloud (gratuit pour repos publics) ou equivalent
  • Quality Gate : 0 bugs, 0 vulnerabilites, duplication < 3%, couverture ≥ 70%
  • Commentaire automatique dans les PR
  • Si SonarCloud impossible (repo prive), expliquez et utilisez un outil local (SonarLint, CodeClimate CLI)

SonarCloud detecte : Bugs, Code Smells, Vulnerabilites, Duplication, Dette technique. C'est votre deuxieme filet de securite apres le linter.

Securite des dependances

Activez au moins un outil de scan de dependances sur votre repo :

Dependabot

Integre a GitHub, gratuit

Snyk

Gratuit pour l'open source

Trivy

Open source, images Docker aussi

Le pipeline doit echouer si une vulnerabilite critique est detectee.

4 Git & Conventions

Evalue sur /2 points

Conventional Commits

Format : <type>(<scope>): <description>
feat(auth): add JWT token validation
fix(api): return 404 when user not found
test(user): add unit tests for UserService
ci: add SonarCloud to pipeline
refactor(db): extract repository pattern
docs: add pipeline architecture diagram
 
BAD: "fix stuff"
BAD: "update"
BAD: "WIP"

Strategie de branches

  • GitHub Flow : main protegee + branches feature
  • Nommage : feature/, fix/, ci/, refactor/
  • Chaque feature → branche → PR → merge
  • Minimum 5 PR dans l'historique

Protection de main : push direct interdit, PR obligatoire, CI verte obligatoire avant merge, branche a jour avec main.

Un historique Git propre fait partie du livrable

Un repo avec 3 commits ("initial commit", "add everything", "fix") montre un manque de rigueur. L'historique doit refleter une progression logique : setup → features → tests → pipeline → docker → iac → monitoring. Minimum 5 Pull Requests mergees.

5 Docker & Conteneurisation

Evalue sur /2 points

Dockerfile

  • Multi-stage build : stage build separe du stage production
  • Image legere : Alpine, slim, ou distroless
  • Utilisateur non-root (securite)
  • .dockerignore : node_modules, .git, .env, tests, docs
  • Aucun secret dans l'image ou le Dockerfile
  • HEALTHCHECK dans le Dockerfile
  • Labels : mainteneur, version

Docker Compose

Si votre app a des dependances (BDD, cache, queue...) :

  • Fichier docker-compose.yml pour le dev local
  • Services : app + BDD + cache si applicable
  • Volumes pour la persistance
  • Reseau dedie
  • Variables d'environnement via .env.example

Optimisation & Securite de l'image

< 200MB

Taille cible de l'image finale

Documentez la taille reelle

Trivy

Scan de vulnerabilites

Integre au pipeline CI

Cache

Layer caching optimise

Ordre des instructions compte

Rappel — Ordre des layers dans un Dockerfile

Les layers qui changent le moins souvent doivent etre en haut (OS, dependances systeme), celles qui changent le plus en bas (copie du code source). Cela optimise le cache Docker et reduit les temps de build.

6 Pipeline CI/CD (GitHub Actions)

Evalue sur /3 points

Pipeline d'Integration Continue (CI)

Se declenche sur chaque push et chaque Pull Request. Voici les etapes attendues :

.github/workflows/ci.yml — Pipeline CI attendu
# Declenchement
on: push + pull_request
 
# ========= JOBS PARALLELES =========
 
[Job 1] Lint & Format check
Verifie que le code respecte les regles
 
[Job 2] Tests unitaires
+ rapport de couverture (seuil ≥ 70%)
 
[Job 3] Tests d'integration
Avec services (BDD en conteneur si besoin)
 
[Job 4] Analyse statique (SonarCloud)
Quality Gate bloquante
 
[Job 5] Scan securite dependances
Echec si vulnerabilite critique
 
# ========= JOBS SEQUENTIELS (needs:) =========
 
[Job 6] Build application (needs: lint, tests)
 
[Job 7] Build image Docker (needs: build)
 
[Job 8] Scan image Docker (Trivy) (needs: docker-build)

Rappel — Parallelisme et dependances

Par defaut, les jobs GitHub Actions tournent en parallele. Utilisez needs: pour creer des dependances entre jobs. Les jobs independants (lint, tests, scan) doivent tourner en parallele pour accelerer le pipeline. N'oubliez pas le cache des dependances (actions/cache) pour accelerer chaque run.

Pipeline de Deploiement Continu (CD)

Auto Staging

  • Deploiement automatique a chaque merge sur main
  • Environnement GitHub Actions : environment: staging

Manuel Production

  • Declenche par workflow_dispatch ou tag/release
  • Environnement GitHub Actions : environment: production avec approval

Si le deploiement reel est impossible

Pas de serveur ? Simulez le deploiement : push de l'image Docker sur GitHub Container Registry (ghcr.io) ou Docker Hub. Le pipeline CD doit etre complet et fonctionnel meme s'il ne deploie que vers un registry. Documentez comment vous deploieriez en production (quel cloud, quelle config, quel cout estime).

Gestion des secrets

  • Secrets via GitHub Secrets (jamais en dur)
  • Acces : ${{ secrets.NOM }}
  • Pas de .env commite
  • Fichier .env.example (sans valeurs) dans le repo

Strategie de rollback

  • Documentee dans le README
  • Mecanisme concret : redeploiement d'un tag precedent, revert de commit, ou rollback d'image Docker
  • Expliquez la procedure etape par etape

Le pipeline DOIT etre vert

Un pipeline systematiquement rouge montre un manque de rigueur. Au moment du rendu, votre pipeline doit etre vert sur la branche main. Un pipeline rouge = points en moins.

7 Infrastructure as Code

Evalue sur /3 points

Utilisez au moins un des deux outils ci-dessous. Les deux = bonus.

⚙

Terraform

Provisionnement d'infrastructure

  • Definir l'infra : serveur, reseau, BDD, DNS, load balancer...
  • State file gere proprement (backend remote ou documente)
  • Variables et outputs utilises
  • Modules si pertinent
  • terraform plan integre au pipeline (sur PR = preview des changements)

Sans cloud ? Utilisez le provider Docker (local), ou LocalStack (simule AWS gratuitement), ou ecrivez les fichiers pour un provider reel et documentez avec des screenshots du terraform plan.

⚙

Ansible

Configuration & deploiement

  • Playbooks : installation dependances, config reverse proxy, deploiement app
  • Inventaire (meme si un seul serveur local ou conteneur Docker)
  • Roles si pertinent
  • Variables et templates (Jinja2)
  • Idempotence : relancer le playbook ne casse rien

Sans serveur distant ? Testez sur un conteneur Docker local ou une VM Vagrant. L'important est de demontrer la maitrise de l'outil, pas d'avoir un serveur cloud.

Terraform vs Ansible : pas le meme role

Terraform = "je cree l'infrastructure" (provisionner une VM, un reseau, un load balancer).
Ansible = "je configure l'infrastructure" (installer des packages, deployer l'app, configurer Nginx).
Les deux sont complementaires. Terraform cree la VM, Ansible la configure.

8 Monitoring & Observabilite

Evalue sur /2 points

Les 3 piliers de l'observabilite

Logs (que s'est-il passe ?), Metriques (comment se porte le systeme ?), Traces (ou est-ce que ca bloque ?). Votre projet doit couvrir au minimum les deux premiers.

Health Check

  • Endpoint /health ou /status
  • Retourne l'etat de l'app ET de ses dependances (BDD, cache, services externes)
  • Utilise par Docker HEALTHCHECK, le load balancer, et le monitoring
GET /health
{
  "status": "healthy",
  "database": "connected",
  "uptime": 12345,
  "version": "1.2.0"
}

Logging structure

  • Logs au format JSON (pas de console.log("ca marche"))
  • Niveaux utilises correctement : debug, info, warn, error
  • Contexte : timestamp, request ID, user ID si applicable
  • Pas de donnees sensibles dans les logs (mots de passe, tokens)
Exemple de log structure
{
  "level": "info",
  "timestamp": "2025-03-15T10:30:00Z",
  "message": "User created",
  "requestId": "abc-123",
  "userId": 42
}

Metriques & Dashboard

  • Prometheus + Grafana ou stack equivalente
  • Metriques de base : requetes/sec, temps de reponse, taux d'erreur, memoire/CPU
  • Dashboard avec minimum 4-5 panneaux pertinents
  • Peut tourner en local via Docker Compose

Si impossible a heberger : faites tourner la stack en local, fournissez des screenshots du dashboard fonctionnel dans la doc.

Alerting

  • Minimum 1 alerte configuree :
  • App down (health check echoue)
  • Taux d'erreur > seuil
  • Temps de reponse degrade
  • Canal : email, Slack, Discord, webhook — au choix

Si impossible a mettre en place : documentez la strategie d'alerting avec les seuils choisis et justifies.

9 Documentation

Evalue sur /1 point

README.md — Le point d'entree de votre projet

Le correcteur lira votre README en premier. Il doit pouvoir comprendre votre projet, vos choix, et lancer le projet en local sans deviner.

  • Description du projet et son objectif
  • Stack technique et justification des choix
  • Comment lancer le projet en local (idealement : docker-compose up)
  • Comment lancer les tests
  • Schema de l'architecture de l'application
  • Schema du pipeline CI/CD (etapes, dependances entre jobs)
  • Badges : build status, couverture, qualite SonarCloud

Documentation des Design Patterns

Pour chaque pattern utilise :

  • Ou il est utilise (fichier, classe, ligne)
  • Pourquoi ici et pas ailleurs
  • Quel probleme il resout concretement
  • Comment il facilite les tests
  • Un schema simple ou diagramme UML (bonus)

Documentation des choix impossibles

Pour chaque exigence non realisee :

  • Ce qui etait demande
  • Pourquoi c'est irrealisable (cout, acces, temps, complexite)
  • Ce qui aurait ete mis en place (outil, configuration, architecture)
  • Le workaround choisi (ou absence justifiee)

Une exigence ignoree sans justification = points perdus. Une explication technique claire = points gagnes.

Dernier rappel

L'IA peut generer du code, un Dockerfile, un pipeline YAML. Ce qu'elle ne peut pas faire a votre place : justifier vos choix dans le contexte specifique de VOTRE projet. Pourquoi ce pattern et pas un autre ? Pourquoi ce seuil de couverture ? Pourquoi cette strategie de deploiement ? C'est la que se fait la difference entre un etudiant qui comprend et un etudiant qui copie-colle.
